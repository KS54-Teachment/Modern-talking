# Технический кейс №3 — Случайное удаление важных данных

**Modern Talking · Центр цифровой поддержки**

## 1. Описание ситуации

Участник проекта случайно удалил важные файлы: часть документации и журнал обращений. Часть файлов была удалена локально и уже закоммичена, часть — удалена только в рабочей копии, но ещё не зафиксирована.

## 2. Анализ и причины

- Невнимательность: удаление файлов/папки без проверки.
- Отсутствие резервных копий и регулярных коммитов.
- Работа напрямую в ветке `main` без отдельных веток и review.
- Команда `rm`/«Shift+Delete» в обход корзины.

## 3. План восстановления

### А. Файл удалён, но изменения не закоммичены
```bash
git restore <файл>          # вернуть из последнего коммита
git checkout -- <файл>     # старый синтаксис
```

### Б. Удаление уже закоммичено
```bash
git log --oneline                 # найти коммит до удаления
git checkout <commit> -- <файл> # вернуть конкретный файл
git revert <commit>               # отменить коммит удаления
```

### В. Потерян целый коммит / ветка
```bash
git reflog                        # история всех действий HEAD
git checkout <hash из reflog>    # вернуться к нужному состоянию
git reset --hard <hash>           # восстановить состояние
```

### Г. Если файл был на GitHub
Повторно клонируйте репозиторий или `git pull origin main` — удалённая копия хранит историю. Также можно открыть файл в истории коммитов на GitHub и восстановить вручную.

### Д. Локальное удаление вне Git
Проверьте корзину/Историю файлов (Windows File History, macOS Time Machine), облачную корзину (OneDrive/Google Drive хранят удалённые файлы 30 дней).

## 4. Меры профилактики

- Коммиты часто и небольшими порциями; пуш на GitHub каждый день.
- Работа через ветки и Pull Request, защита ветки `main` (branch protection).
- Резервное копирование по правилу 3-2-1.
- Включить корзину и историю версий файлов.
- Проверять перед удалением, не использовать «Shift+Delete» без необходимости.

## 5. Вывод

Большинство случаев случайного удаления решаются без потерь, если проект ведётся в Git и регулярно пушится на GitHub. Главная защита — резервные копии и дисциплина коммитов.
